**概要**

- **目的:** ユーザが迷わず、ストレスなくMarkdownを読み書きできる、2026年時点の先端UX設計をまとめる。
- **ターゲット:** コンテンツ制作者（個人・チーム）、技術文書作成者、コラボレーション利用者。

**設計原則**

- **予測可能性:** 操作結果を常に明示（状態表示・トースト・アニメ）。
- **最小認知負荷:** 必要な操作のみ目立たせ、上級機能は段階的開示。
- **即時フィードバック:** 自動保存、差分ハイライト、ライブプレビュー。
- **コンテキスト保護:** ローカルWIP、バージョン履歴、衝突解決の明示的UI。
- **アクセシビリティ:** キーボード優先、スクリーンリーダー対応、高コントラスト/フォント調整。

**ページ構成（デスクトップ）**

- **ヘッダー（狭）:** ページタイトル、公開/共有ステータス、重要アクション（保存/公開）。
- **メインペイン:** 左：エディタ（折りたたみ可）／右：プレビュー（同期スクロール）。
- **右サイドバー:** メタデータ・目次・テンプレート・ブロック挿入パネル。
- **ボトムスナックバー:** 保存状態、アクティブブランチ/バージョン、共同編集者の表示。

**モバイル構成（単一カラム）**

- **フローティングツールバー:** 画面下に主要アクション（書式・添付・プレビュー切替）。
- **切替UX:** 明確なタブで「編集」「プレビュー」「閲覧モード」を切り替え。

**主要コンポーネント仕様**

- **エディタ（ハイブリッド）:** ソースエディタとブロックベースWYSIWYGを統合。
    - **ブロック単位:** 見出し/段落/コード/画像/埋め込みをブロック化。ドラッグで並べ替え、右側でプロパティ編集。
    - **コマンドパレット:** `⌘K` で挿入/変換/テンプレ検索。
    - **キーボードファースト:** Tabでインデント、`Ctrl/Cmd+Enter`でプレビュー切替、`Ctrl/Cmd+S`で明示保存（自動保存はバックグラウンド）。
- **プレビュー:** 差分レンダリング・同期スクロール。編集中ブロックはプレビュー上でハイライト。
- **ツールバー:** コンテキスト依存で最小表示。選択に応じて変化（リンク挿入/コードフォーマット等）。
- **テンプレート/スニペット:** カテゴリ化されたサイドバー、ドラッグ＆ドロップで挿入。
- **バージョン履歴UI:** タイムライン表示＋行/ブロック差分ビュー。復元・比較・ブランチ作成を提供。
- **コラボレーション (排他制御):** 同時リアルタイム編集（CRDT/OT）は採用しません。編集は排他ロック方式を採用します。
    - 編集中のメモはロックされ、他ユーザーは読み取り専用で表示されるか、編集申請を行えます。

**重要操作フローとエッジケース**

**同時編集衝突:** 本プロジェクトではリアルタイム自動マージを行いません。排他ロック取得の上で編集を行い、もしロック状態でない更新が行われた場合は `409 Conflict` を返し、クライアントに差分確認（手動マージ）を要求します。

**アクセシビリティ & 多言語対応**

- **キーボード全対応:** 全操作にショートカット経路。ショートカット一覧モーダルでカスタム可能。
- **スクリーンリーダー:** ARIAランドマーク、ブロックごとのラベル、live-regionで差分通知。
- **表示調整:** コントラスト切替、フォントサイズ・行間調整。日本語向けルビや折り返しオプション。
- **ローカライズ:** UI文言は外部辞書化、右から左言語対応を確保。

**パフォーマンスとオフライン**

- **差分レンダリング:** 長文でも仮想化＋最小再描画でスムーズ。
- **資産最適化:** 画像は遅延読み込み・プログレッシブ。エディタのソースは分割ロード。
- **オフライン編集:** 編集用の基本資産をService Workerでキャッシュ、WIPはIndexedDBに保持。

**開発用ハンドオフ（要件）**

- **Design tokens:** 色・間隔・フォントを変数化（例: `--ui-primary`, `--space-4`）。
- **API 要求:**
    - `PATCH /pages/:id` は部分パッチを受け取り、楽観的更新をサポート。
    - `POST /pages/:id/attachments` は非同期アップロードレスポンス（asset id, temp URL）。
    - `GET /pages/:id/history` はタイムラインと差分を返す。
    - リアルタイムはWebSocket/RTCでCRDTトランスポートを推奨。
- **QAシナリオ:** 自動保存中断→再接続、同時編集衝突、画像アップロード中断、スクリーンリーダー操作フロー。

**ワイヤーフレーム指示（デザイナー向け）**

- **ブレークポイント:** Desktop (>=1200px)、Tablet (768–1199px)、Mobile (<768px)。
- **必須画面:** 編集（エディタ＋プレビュー）、プレビュー専用、バージョン比較、共有/公開設定。
- **注記:** 各コンポーネントはFigmaでコンポーネント化（EditorBlock, PreviewPanel, VersionTimeline, CommandPalette）。

**MVP優先度（実装順）**

1. 自動保存（ローカルWIP + サーバ同期）
2. ソースエディタ＋ライブプレビュー（同期スクロール）
3. 画像アップロード（非同期、プログレス）
4. バージョン履歴表示（簡易タイムライン＋差分）
5. 基本アクセシビリティ（キーボード・ARIA）

**付録 — キーボードショートカット（提案）**

- `Ctrl/Cmd+S`: 明示保存（オプション）
- `Ctrl/Cmd+Enter`: 編集↔プレビュー切替
- `⌘K`: コマンドパレット
- `Ctrl/Cmd+Z` / `Ctrl/Cmd+Shift+Z`: Undo/Redo

### 編集ロックの失効時の UX（推奨仕様）

編集中に排他ロックが期限切れ（TTL切れ）になった場合の UI 挙動を明確化します。実装者は以下の状態遷移と API 呼び出し方針に従ってください。

状態:

- `Editing` — ロック保有中。通常編集可。クライアントは `LOCK_TTL_SECONDS` の半分ごとに `PATCH /api/notes/:id/lock` を呼びリフレッシュを試みる。
- `NearExpiry` — 残り 30 秒（configurable）を切ると UI は警告トーストを表示（例: 「ロックがまもなく切れます — 自動更新中」）。
- `RefreshFailed` — リフレッシュのネットワーク失敗やサーバーエラーが続く（例: 3 回再試行失敗）と、クライアントは非侵襲な警告を表示し **内部的には編集中の WIP をローカル（IndexedDB）に保存**する。編集領域は引き続き編集可能だが、保存ボタンは「ローカルに保存（コピーを作成）」にフォールバックして、サーバ上データを上書きしない挙動にする。
- `LockLost` — サーバが `409 CONFLICT`（別ユーザーが取得） を返した場合、編集領域は読み取り専用に切り替え、画面上にロック所有者情報（`owner.username` と残りTTL）を表示する。ユーザには次の選択肢を提示する:
    1. **再取得を試みる**（`POST /api/notes/:id/lock` を呼ぶ。成功すれば編集再開）
    2. **別名で保存 (Save as copy)** — 現在の WIP を新規メモとして保存する（`POST /api/notes` を呼び、ユーザーに新しいメモ URL を提示）
    3. **読み取りモードのまま続ける**（変更を破棄しないがサーバに保存されない）

保存時の挙動:

- ユーザーが `保存` を押した時点でクライアントは現在のロック状態を検証する（`If-Lock` ヘッダ、または事前に `GET /api/notes/:id/lock` を確認）。
- ロック有効: 通常通り `PATCH /api/notes/:id` を許可。
- ロック無効: クライアントは上記ダイアログで「別名で保存」か「再取得を試行」かを選ばせる。自動的に強制上書きするのは禁則（データ消失防止）。

UI 表示例（日本語）:

- NearExpiry トースト: 「このノートのロックがあと 25 秒で切れます。自動更新を試行中…」
- RefreshFailed 警告: 「ロックの更新に失敗しました。作業が失われないようローカルに保存しました。」（ボタン: 「再試行」「保存してコピー」）
- LockLost モーダル: 「他のユーザー（`bob`）が編集中のため編集できません。ロックを取得できれば編集を再開できます。」（ボタン: 「再取得」「別名で保存」「閲覧のみ」）

実装メモ:

- インデックスDB（IndexedDB）へ WIP を保存する仕組みは既に自動保存の一部として必須にする。ローカル保存は `pageId + draftId` で管理。
- クライアントはロックのリフレッシュを行う際、サーバのレスポンスコードを厳密に扱う: `200` は成功（expiresAt 更新）、`401/403` は認証/権限の再確認、`409` は別所有者による拒否。
- 自動リフレッシュの頻度と再試行ポリシーは環境変数 `LOCK_TTL_SECONDS` と `LOCK_REFRESH_RETRY`（回数）で調整可能にする。

この仕様により、編集中のユーザーが TTL 切れで突然データを失うリスクを低減し、UX の一貫性を担保します。

    - 左にソース/ブロックエディタ、右にプレビューが表示される。左右はドラッグでリサイズ可能。

    - 指定した時刻のスナップショットを選んで差分プレビューできる。復元とブランチ作成が可能である。

### API スキーマ（MVP 用）

以下は最小限のAPI定義。詳細はセキュリティ・認証ルールで補う。

1. ページ取得

- GET /api/pages/:id
    - Response 200
    - body: { id, title, body: string, blocks: [{id,type,props}], meta: {status,author,updatedAt} }

2. 部分更新（パッチ）

- PATCH /api/pages/:id
    - Request body (JSON Patch / custom): { patch: [{ op: 'replace'|'add'|'remove', path: '/blocks/3', value: ... }] }
    - Response 200: { ok: true, revisionId }
    - サーバは楽観的ロック用の `revisionId` を返す。

3. アタッチメントアップロード

- POST /api/pages/:id/attachments
    - multipart/form-data upload
    - Response 202: { assetId, uploadUrl, status: 'processing' }
    - 完了通知は Webhook または WebSocket/WS イベントで行う。

4. 履歴取得

- GET /api/pages/:id/history?limit=50
    - Response: [{ revisionId, author, timestamp, summary }]

5. リアルタイム同期（設計）

- WebSocket または WebRTCデータチャネルで CRDT ops を交換。
- 初回接続: サーバは現在のCRDT状態 + revisionId を送信。

## 高度化設計 — 2026年最先端のUXディテール

以下は、既存仕様を次の段階へ押し上げるための具体的／実装寄りの拡張点です。ユーザの迷いを徹底排除し、感覚的に洗練された体験を目指します。

### 1) マイクロインタラクションとモーションポリシー

- **保存トランザクション:** 明確な状態変化（保存中：淡いドットアニメ／保存済み：短いバッジフェード）。状態は3段階（idle→pending→synced）で正確に表出。
- **入力フィードバック:** 1文字以上のコミットでインライン軽微アニメ（行リフト）を入れ、視覚的に変更箇所を把握しやすくする。
- **ドラッグ操作:** ブロック移動時は影の強調と挿入位置のプレースホルダが滑らかにトランジションする（0.12s easing）。
- **遅延アニメーション:** 長時間表示される要素（履歴差分、マージUI）のアニメは短く、ユーザ操作の阻害を避ける。

### 2) 高度なツールバー & コンテキストUI

- **インテリジェント・ミニツールバー:** テキスト選択時に最小限の頻出アクション（太字・リンク・コード・インラインコメント）を浮上表示。選択種類で優先ボタンが変わる（見出し選択なら階層切替）。
- **ブロックヘッダー:** 各ブロックの左にハンドル＋コンテキストメニュー（変換/複製/スタイル）を常時表示し、一貫した発見性を確保。
- **コマンドサジェスト:** コマンドパレットにAI補完を組み込み、自然言語で「この段落を要約してタイトルを作成して」といった操作が可能。

### 4) 発見性・学習設計（オンボーディングの強化）

- **漸進的開示:** 初回ユーザには最小UI（コアのみ）を提示、利用頻度に応じて拡張機能をアンロック（テンプレ、細かいフォーマット等）。
- **コンテキストヘルプ:** 初回操作で軽いヒントバブルを表示（閉じると履歴に保存）。いつでも `?` キーでその文脈のヘルプを呼び出せる。

### 5) デザインシステム：コンポーネント詳細とVariant設計

- **EditorBlock（Variants）:** `type` (paragraph, heading, code, image), `state` (editing, collapsed, focused), `decor` (callout, quote) を variant として定義。
- **Auto-layout 指針:** コンポーネントは横方向は `padding: var(--space-3)`、内部は `gap: var(--space-2)`。FigmaではAuto Layout必須。最小幅（block min-width）を定義してレスポンシブ崩れを防止。
- **Iconography:** 16/20/24px の3サイズを用意し、アクセシビリティのために最低2:1のタップターゲットを保証。

### 6) テーマとダークモード戦略

- **色パレット:** `surface`/`muted`/`accent`をRGBとHSL両方で定義し、ダークでの色反転を計算的に生成（アクセシビリティの保証）。
- **モード切替:** 即時切替＋保存（ユーザ設定）／OS設定に追従オプション。

### 7) アクセシビリティ拡張（詳細）

- **フォーカスの可視化:** カスタムフォーカスリング（2px）を採用、`prefers-reduced-motion` に従う。
- **読み上げ戦略:** 編集カーソル移動時は `aria-live="polite"` で「見出しレベル3、段落」など簡潔に通知。差分通知は `assertive` にしない。
- **キーボードフロー:** 主要パネル（エディタ／プレビュー／サイドバー）間のジャンプは `Ctrl/Cmd+1/2/3` を割り当てる。

### 8) 同時編集と競合のUX（詳細）

- **排他ロック方式:** リアルタイムCRDTは採用しません。編集前に `POST /api/notes/:id/lock` でロックを取得し、取得者のみが編集可能となります（他者は読み取り専用）。
- **ロック表示:** UI はロック所有者の表示（名前、アバター）と残りTTLを明示します。長時間作業する場合はクライアント側でリフレッシュを行い TTL を延長します。
- **競合時の取り扱い:** ロックが取れていない状態で更新が試行された場合は `409 Conflict` を返却し、クライアントに差分確認／手動マージUIを表示します。自動による文書破壊は行いません。

### 9) セキュリティ・プライバシー設計ノート

- **AI送信ポリシー:** 送信前に明確な同意を要求。企業ユーザ向けにオン/オフ、ログ抑止、リージョン制限オプションを設ける。
- **添付ファイルの保護:** 一時URLは短命にし、署名付きURLを使用。アクセス権限はページ権限に紐付け。

### 10) パフォーマンス目標と計測

- **First Input Delay（FID）**: < 50ms 目標
- **Time to Interactive（TTI）**: エディタシェルは < 1s（キャッシュを活用）
- **レンダリング予算:** 1回の差分レンダリングは 16ms 未満を目指す。

### 11) 開発ハンドオフ追加項目

- **Storybook stories:** `EditorShell`, `BlockEditor`, `Toolbar`, `PreviewPane`, `VersionTimeline` の各コンポーネントで少なくとも `default` / `focused` / `error` / `with-data` を作成。
- **Figma tokens ファイル:** 色・間隔・タイポを `tokens.json` としてエクスポートし、フロント側で `style-dictionary` に取り込む。
